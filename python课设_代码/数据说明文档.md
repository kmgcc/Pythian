# 数据说明文档

## 一、项目数据总览

本项目涉及三类外部数据源和若干程序内部生成的中间数据，最终汇聚为一个用于机器学习训练的主数据集。

| 数据类别 | 来源平台 | 文件/接口 | 大小 |
|---------|---------|----------|------|
| 天光光谱实测数据 | Zenodo（SKYSPECTRA 数据集） | 5 个 CSV 文件下载 | ~75 MB |
| 历史逐小时天气数据 | Open-Meteo Archive API | REST 接口 + 本地 JSON 缓存 | 按需请求 |
| 商业 LED 实测光谱 | Harald Brendel 个人主页 | 1 个 CSV 文件下载 | 88 KB |
| 训练数据集（最终产物） | 项目内部生成 | `data/real_spectrum_weather_dataset.csv` | 5,664 行 × 53 列 |

---

## 二、数据来源详解

**平台**：Zenodo（欧洲开放科学数据中心），记录编号 `8147546`

**内容**：全球 5 个观测站在真实天气条件下，使用光谱辐射计测量的水平面光谱辐照度数据，以及配套的气象和太阳位置元数据。

### 2.2 Open-Meteo 历史天气 API

**平台**：Open-Meteo（open-meteo.com），免费开放的气象数据服务

**接口**：`https://archive-api.open-meteo.com/v1/archive`

**用途**：SKYSPECTRA 数据集的 `meta_weather.csv` 只提供了天空类型和照度/辐照度，缺少云量百分比、湿度、温度、降水等细粒度气象参数。Open-Meteo 作为补充数据源，按观测站经纬度和日期范围拉取逐小时历史天气。

#### 请求参数

| 参数 | 说明 |
|------|------|
| latitude / longitude | 观测站经纬度 |
| start_date / end_date | 日期范围（ISO 格式） |
| hourly | `cloud_cover,relative_humidity_2m,temperature_2m,precipitation` |
| timezone | `auto`（自动匹配当地时区） |

#### 返回字段

| 字段 | 说明 | 单位 | 项目中的处理 |
|------|------|------|------------|
| cloud_cover | 总云量 | % | 除以 100 转为 0–1 |
| relative_humidity_2m | 2 米处相对湿度 | % | 除以 100 转为 0–1 |
| temperature_2m | 2 米处气温 | ℃ | 直接使用 |
| precipitation | 降水量 | mm/h | 直接使用 |

#### 缓存策略

- 缓存目录：`data/external/weather_cache/`
- 文件命名：`weather_{纬度}_{经度}_{开始日期}_{结束日期}.json`
- 首次请求从 API 下载并缓存，后续直接读取本地 JSON
- 重试机制：最多 3 次，每次间隔 2 秒

### 2.3 商业 LED 实测光谱数据

**平台**：Harald Brendel 个人主页（haraldbrendel.com）

**文件**：`led_spd_350_700.csv`（88 KB）

**内容**：29 个商业 LED 的实测光谱功率分布（SPD），波长范围 350–700nm，2nm 步长（共 176 个波长点）。

**用途**：从中选取特定样本构造项目所需的 LED 通道光谱：

| 用途 | 选取的样本 | 构造方法 |
|------|-----------|---------|
| 5 个窄带彩色通道（深蓝/蓝光、青光、绿光、琥珀光、红光） | 从 29 个 LED 中匹配峰值波长最接近目标的窄带 LED | `select_measured_color_led()` 按峰值距离筛选 |
| 2700K 暖白通道 | Row 21（蓝光泵浦）+ Row 11（宽谱荧光粉）+ Row 5（暖红荧光粉） | 按比例混合：`0.08×蓝 + 0.55×宽谱 + 0.75×暖红` |
| 6500K 冷白通道 | 同上三个样本 | 按比例混合：`1.2×蓝 + 0.4×宽谱 + 0.05×暖红` |

**输出文件**：`data/dual_white_led_spectrum.csv`（41 行 × 3 列：wavelength_nm, warm_2700k, cool_6500k）

## 四、数据处理流水线

整个数据从原始下载到可用于训练，经过以下 8 个步骤（由 `src/real_data_pipeline.py` 执行）：

```
Step 1: 从 Zenodo 下载 5 个 CSV → data/external/
            ↓
Step 2: 解析观测站经纬度（清洗 N/E/W/S 方向字符）
            ↓
Step 3: 合并 meta_weather + meta_sun_positions（内连接，3 键匹配）
            ↓
Step 4: 光谱数据 pivot（长→宽）+ 重采样（1nm→10nm，380-780nm）
            ↓
Step 5: 元数据与光谱数据内连接（按 location_code + timestamp）
            ↓
Step 6: 调用 Open-Meteo API 获取逐小时天气，按最近小时对齐
            ↓
Step 7: 映射天气分类（晴/多云/阴/雨）
            ↓
Step 8: 输出 real_spectrum_weather_dataset.csv
```

### 关键处理细节

1. **光谱重采样**：原始数据为 350–700nm / 1nm 步长（351 个点），通过 `numpy.interp` 线性插值到 380–780nm / 10nm 步长（41 个点）
2. **天气对齐**：光谱测量是每 10 分钟一次，天气数据是逐小时一次，通过 `dt.round("h")` 取最近小时匹配
3. **数据过滤**：`dropna(subset=[所有特征列 + 41 个光谱列])`，任何特征缺失的样本被丢弃
4. **照度缺失问题**：原始 SKYSPECTRA 数据集中，北京/柏林/格拉纳达的 `global_horizontal_illuminance` 全部为空，导致这三个站点的数据在 `dropna` 时被全部过滤

## 五、训练数据的构建方式

### 5.1 输入特征（X）

模型使用 8 个气象特征作为输入：

| 特征 | 类型 | 预处理 |
|------|------|--------|
| hour | 连续 | StandardScaler 标准化 |
| cloud_cover | 连续 | StandardScaler 标准化 |
| humidity | 连续 | StandardScaler 标准化 |
| temperature | 连续 | StandardScaler 标准化 |
| precipitation | 连续 | StandardScaler 标准化 |
| solar_altitude | 连续 | StandardScaler 标准化 |
| outdoor_lux | 连续 | StandardScaler 标准化 |
| weather | 类别 | OneHotEncoder 编码（晴/多云/阴/雨 → 4 列） |

预处理器由 `sklearn.compose.ColumnTransformer` 构建，连续特征走 `StandardScaler`，类别特征走 `OneHotEncoder`。

### 5.2 目标变量（Y）

目标是 41 维的相对光谱曲线（`wavelength_380` ~ `wavelength_780`）。

**降维处理**：直接预测 41 维输出效率低且容易过拟合，因此先对目标变量做降维：

1. **StandardScaler**：对 41 维光谱做标准化（均值为 0，方差为 1）
2. **PCA**：将标准化后的 41 维光谱降至 **5 个主成分**

PCA 保留的累计解释方差比例通常在 95% 以上，即 5 个主成分已经捕获了光谱曲线绝大部分的变化信息。

训练时模型学习的是 `X → PCA 成分` 的映射；预测时先输出 5 维 PCA 成分，再通过 `pca.inverse_transform()` 和 `target_scaler.inverse_transform()` 还原为 41 维光谱。

### 5.3 数据集划分

- **训练集 / 测试集**：80% / 20%（`test_size=0.2`）
- **划分方式**：`stratify=df["weather"]`，按天气类别分层抽样，确保训练集和测试集中四种天气的比例一致
- **随机种子**：`random_state=42`，保证结果可复现

### 5.4 模型训练

4 个回归模型同时训练，每个模型的 Pipeline 结构为：

```
输入特征 → [StandardScaler + OneHotEncoder] → [回归模型] → 5 维 PCA 成分
                                                              ↓
                                            [PCA 逆变换 + StandardScaler 逆变换]
                                                              ↓
                                                        41 维预测光谱
```

| 模型 | 核心超参数 |
|------|-----------|
| Linear Regression | 无（默认配置） |
| KNN Regression | n_neighbors=12, weights="distance" |
| Decision Tree | max_depth=14 |
| Random Forest | n_estimators=180, max_depth=18, min_samples_leaf=2, n_jobs=-1 |

### 5.5 评价指标

| 指标 | 公式含义 | 关注点 |
|------|---------|--------|
| MAE（平均绝对误差） | 预测值与真实值之差的绝对值的平均 | 整体偏差水平 |
| RMSE（均方根误差） | 预测值与真实值之差的平方的平均再开根号 | 对大误差更敏感，排序依据 |
| R²（决定系数） | 模型解释了多少比例的方差 | 1 = 完美预测，0 = 不如用平均值 |

模型选择标准：按 RMSE 升序排列，取 RMSE 最低的模型作为最优模型（`best_model`）。

## 六、辅助数据文件

### 6.1 天然日光参考光谱

- **文件**：`data/natural_daylight_reference.csv`
- **内容**：41 行 × 2 列（wavelength_nm, relative_intensity）
- **生成方式**：从训练数据集中筛选天气为"晴"且太阳高度角 ≥ 40° 的样本，对其 41 维光谱取均值并归一化
- **用途**：作为 LED 补偿的目标光谱（补偿算法的参考基准）

### 6.2 双色温 LED 光谱

- **文件**：`data/dual_white_led_spectrum.csv`
- **内容**：41 行 × 3 列（wavelength_nm, warm_2700k, cool_6500k）
- **生成方式**：从 Harald Brendel 的 29 个实测 LED 中选取蓝光泵浦、宽谱荧光粉、暖红荧光粉三个样本，按比例混合构造 2700K 暖白和 6500K 冷白通道

### 6.3 模型文件

- **文件**：`models/spectrum_model.joblib`
- **内容**：`ModelTrainingResult` 数据类的序列化，包含训练好的 Pipeline、PCA 模型、StandardScaler、测试集预测结果等
- **格式**：joblib（基于 pickle 的序列化）

## 七、数据质量说明

### 已知问题

1. **城市分布极不均衡**：巴黎占 99.5%，新加坡仅 0.5%，北京/柏林/格拉纳达的数据因照度字段缺失而完全丢失
2. **天气分布不均衡**：晴天占 55.7%，多云仅 6.0%
3. **照度估算**：Web 应用中的室外照度使用随机森林回归估算（`cached_outdoor_lux_model`），而非直接从数据集读取，因为部分场景下需要估算用户自定义输入条件的照度

### 数据完整性

- 所有 5,664 条样本的 53 个字段均无缺失值
- 光谱值均为非负浮点数，范围 [0, 1]（归一化后）
- 天气 API 数据通过本地 JSON 缓存，避免重复网络请求
